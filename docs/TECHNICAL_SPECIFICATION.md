## 2. Цели проекта

### 2.1. Бизнес-цели

1. **Рост LTV** через систему автоматического возврата пациентов («ведение»).
2. **Контроль качества** работы кураторов — прозрачность задач, своевременность.
3. **Увеличение среднего чека** через кросс-продажи и пакетные предложения.
4. **Снижение потерь** пациентов за счёт дожима и реактивации базы.

### 2.2. Целевые KPI (через 6 месяцев после запуска)

| Метрика | Цель |
|---------|------|
| Конверсия «Ведение → Возврат» | ≥ 55% |
| % пустых финальных консультаций | 0% |
| Задачи выполнены в срок | ≥ 95% |
| Средний чек повторного визита | ≥ 95% от первичного |
| No-show при авто-напоминаниях | < 10% |
| LTV реактивированных | ≥ 80% от первичных |

## 5. As-Is / To-Be процессы

### 5.1. AS-IS: Текущее состояние

**Проблемы:**
- Кураторы пишут напоминания в свободные поля «Комментарии» → они теряются.
- Нет единого контроля задач → переработки 40–60 ч/мес.
- Нет базы лидов → необработанные обращения исчезают.
- Распределение пациентов — вручную, устно → «война» за иногородних.
- Финальная консультация может закрыться без фиксации ведения → «пустые финалки».
- Отсутствует отдельная роль для звонарей-помощников → кураторы перегружены.
- Суммы в CRM и 1С расходятся (~1,8 млн руб. по одному городу) → нет источника истины.

### 5.2. TO-BE: Целевое состояние

| Процесс | AS-IS | TO-BE |
|---------|-------|-------|
| Запись на приём | Вручную / телефон | Заказ в CRM с источником обращения и тегами |
| Передача пациента врач→куратор | Сразу / словесно | Автозадача через +3 рабочих дня |
| Работа куратора | Комментарии в карточке | Структурированные задачи с метриками |
| Помощники | Нет системного места | Специализированное рабочее место |
| Финальная консультация | Может закрыться без данных | Блокировка до заполнения обязательных полей |
| Возврат пациента | Устная договорённость | Автоматический сценарий дожима 0–3–7–14–30 |
| Контроль абонементов | Вручную по памяти | Авто-задача на предпоследнем занятии |
| Дашборд РОП | Excel / нет | Дашборд в 1С: просрочки, сумма, по кураторам |

## 14. Миграция данных

### 14.1. Что мигрируется
- Нет исторических данных для миграции из сторонних систем (CRM ранее не было).
- Теги и сегменты пациентов — проставляются вручную при первичной работе.
- Данные о пациентах, услугах, оплатах — уже существуют в 1С:Медицина.

### 14.2. Инициализация справочников
- Источники обращений — заполняются администратором до запуска.
- Типы ведения — заполняются по утверждённому списку.
- Привязка кураторов к помощникам — настраивается в разделе «Настройки CRM».

> OQ-006: Алгоритм очистки устаревших услуг без потери истории — ответ Наташа + IT до 20.06.2026.

## 16. Риски и ограничения

| Риск | Вероятность | Влияние | Митигация |
|------|------------|---------|----------|
| Позиция главврача по электронному листу | Высокая | Блокировка Фазы 2 (2.19) | На запуске — ручной ввод данных куратором |

| Нестабильность API мессенджеров | Средняя | Сбои дожима | Retry-логика; fallback на задачу куратору |
| Несоблюдение регламентов врачами | Высокая | «Пустые финалки» без системы | Блокирующая валидация (FR-008) |
