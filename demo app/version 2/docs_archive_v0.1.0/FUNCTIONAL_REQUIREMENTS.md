## FR-001: Карточка пациента

| Поле | Значение |
|------|----------|
| **ID** | FR-001 |
| **Название** | Единый профиль пациента |
| **Описание** | Система ведёт единый профиль пациента с полной хронологией визитов, заказов, задач, оплат и коммуникаций. Повторный пациент не создаётся заново — открывается новый заказ в существующем профиле. |
| **Бизнес-ценность** | Исключает дублирование, обеспечивает историю 360° для куратора. |
| **Роли** | Все роли (куратор, помощник, РОП, врач, администратор) |
| **Приоритет** | Must |
| **Preconditions** | Пациент обратился в клинику (по телефону, лично, через сайт). |
| **Main Flow** | 1. Администратор / куратор создаёт профиль пациента. 2. Система проверяет дублирование по ФИО + телефон + ДР. 3. При совпадении — предлагает открыть существующий профиль. 4. При создании нового заказа — привязывается к профилю. |
| **Alternative Flows** | AF-001-1: Дубликат найден — система показывает предупреждение и предлагает объединить. |
| **Validation Rules** | ФИО — обязательно; Телефон — обязательно, формат +7XXXXXXXXXX; ДР — обязательно для идентификации дубликатов. |
| **1С Metadata Objects** | Справочник `ФизическиеЛица` (типовой) + расширение реквизитами CRM; РегистрСведений `ИсторияКонтактовCRM`. |
| **Acceptance Criteria** | **Given** профиль пациента создан; **When** куратор вводит совпадающие ФИО+телефон; **Then** система предупреждает о дубликате и не создаёт второй профиль. |
| **Estimate** | 44 ч (см. WORK_ESTIMATION.md, задача 1.1) |

## FR-002: Сущность «Заказ»

| Поле | Значение |
|------|----------|
| **ID** | FR-002 |
| **Название** | Заказ как базовая единица работы с пациентом |
| **Описание** | Вместо «Сделки» используется «Заказ». Заказ содержит: тип приёма (первичный/повторный/промежуточный/финальная консультация), услуги, оплату, ответственного куратора, задачи, статус. Один пациент может иметь несколько заказов. |
| **Бизнес-ценность** | Соответствует терминологии 1С:Медицина. Разграничивает циклы лечения. |
| **Роли** | Куратор, помощник куратора, врач |
| **Приоритет** | Must |
| **Preconditions** | Профиль пациента существует. |
| **Main Flow** | 1. Куратор создаёт заказ из профиля пациента. 2. Указывает тип приёма. 3. Добавляет услуги (из справочника 1С). 4. Назначает ответственного куратора. 5. Сохраняет. |
| **Alternative Flows** | AF-002-1: Врач предзаполняет заказ услугами из консультативного листа (Фаза 2). |
| **Validation Rules** | Тип приёма — обязателен. Хотя бы одна услуга или тип «ортопедия» (исключение). Ответственный куратор — обязателен. |
| **1С Metadata Objects** | Документ `ЗаказCRM`; Перечисление `ТипыПриёмовCRM`; РегистрСведений `СтатусыЗаказовCRM`. |
| **Acceptance Criteria** | **Given** открыт профиль пациента; **When** куратор нажимает «Создать заказ»; **Then** создаётся заказ с обязательными полями, привязанный к профилю. |
| **Estimate** | 44 ч |

## FR-003: Воронка заказов и статусы

| Поле | Значение |
|------|----------|
| **ID** | FR-003 |
| **Название** | Воронка продаж с управляемыми статусами |
| **Описание** | Заказ проходит через статусы воронки: 1. Записан → 2. Пришёл → 3. Приём проведён → 4. Сделано предложение → 5. Оплачено частично → 6. Оплачено полностью → 7. В процессе лечения → 8. Завершён → 9. Отказ. Дополнительный статус: «МРТ в динамике» (для отложенных циклов). |
| **Бизнес-ценность** | Прозрачность процесса. Управление воронкой. Основа для дашборда РОП. |
| **Роли** | Куратор, помощник, РОП |
| **Приоритет** | Must |
| **Preconditions** | Заказ создан. |
| **Main Flow** | 1. Куратор переводит заказ в следующий статус. 2. Система фиксирует время перехода. 3. При переходе в отдельные статусы — автоматически создаются задачи (см. FR-010). |
| **Alternative Flows** | AF-003-1: Заказ «МРТ в динамике» — не включается в текущую конверсию воронки. |
| **Validation Rules** | Паспорт процесса (OQ-009) определяет допустимые переходы. Статус «Завершён» — только если проведена финальная консультация или прошли все услуги. |
| **1С Metadata Objects** | Перечисление `СтатусыЗаказаCRM`; РегистрСведений `ИсторияСтатусовЗаказов`. |
| **Acceptance Criteria** | **Given** заказ в статусе «Сделано предложение»; **When** куратор переводит в «Оплачено частично»; **Then** фиксируется время перехода, создаётся задача куратору по расписанию дожима. |
| **Estimate** | 30 ч |

## FR-006: Система задач кураторов

| Поле | Значение |
|------|----------|
| **ID** | FR-006 |
| **Название** | Структурированные задачи вместо комментариев |
| **Описание** | Каждое напоминание — это задача с: датой, типом, ответственным, статусом (открыта/выполнена/просрочена/пропущена). Задачи создаются автоматически по триггерам или вручную. При закрытии задачи — обязательный текстовый комментарий. Быстрое закрытие (без комментария) — только для типа «Ортопедия». |
| **Бизнес-ценность** | Контроль работы кураторов. Аудит коммуникаций. Исключение потери напоминаний в свободном тексте. |
| **Роли** | Куратор, помощник куратора, РОП |
| **Приоритет** | Must |
| **Main Flow** | 1. Система создаёт задачу (авто или вручную). 2. Куратор / помощник видит задачу в рабочем месте. 3. Выполняет действие (звонок, сообщение). 4. Закрывает задачу с комментарием. 5. РОП видит метрику: своевременно / с опозданием / пропущено. |
| **Validation Rules** | Закрытие задачи без комментария — запрещено (кроме тега «Ортопедия»). Обязательный минимум комментария: 5 символов. |
| **1С Metadata Objects** | Документ `ЗадачаCRM`; Перечисление `ТипыЗадачCRM`; РегистрСведений `МетрикиЗадачCRM`. |
| **Acceptance Criteria** | **Given** куратор пытается закрыть задачу без комментария; **When** нажимает «Закрыть»; **Then** система блокирует закрытие и выводит сообщение «Введите результат выполнения». |
| **Estimate** | 16 ч (задача 1.26) + 22 ч (задача 1.27) |

## FR-011: Роль «Помощник куратора»

| Поле | Значение |
|------|----------|
| **ID** | FR-011 |
| **Название** | Новая роль: помощник куратора |
| **Описание** | Помощник куратора — постоянный сотрудник (или временно назначенный куратор). Закреплён за конкретными кураторами. Функции: обзвон по заданиям кураторов, изменение даты следующего контакта, составление расписания, закрытие задач с комментарием, реанимация базы. | **Бизнес-ценность** | Снижение нагрузки на кураторов (≥80–100 звонков/день). Реанимация «потерянной» базы. |
| **Роли** | Помощник куратора |
| **Приоритет** | Must |

| **1С Metadata Objects** | Роль `ПомощникКуратораCRM`; РегистрСведений `КуратораПомощника`; ОтборыПоРоли. |
| **Acceptance Criteria** | **Given** помощник авторизован; **When** открывает список задач; **Then** видит только задачи кураторов, за которыми закреплён; ДМС-пациенты скрыты; пациенты с «Не связываться» недоступны для закрытия. |
| **Estimate** | 158 ч (блок 1C) |
