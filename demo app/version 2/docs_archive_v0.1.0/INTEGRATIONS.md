## 1. Интеграция с 1С:Медицина.Поликлиника (основная)

### 1.1. Архитектура

CRM реализован как расширение конфигурации (.cfe). Данные 1С:Медицина доступны напрямую через механизмы платформы — подписки на события, общие модули, запросы к типовым регистрам.

**Преимущества подхода:**
- Нет сетевых задержек (работа в одной базе).
- Доступ ко всем объектам типовой конфигурации без API.
- Обновления типовой конфигурации не разрушают расширение (при соблюдении правил расширений 1С).

> **Предположение (OQ-005):** Архитектура подтверждена, если CRM работает в той же информационной базе. Если требуется отдельная база — необходим HTTP-сервис. Решение ожидается от Вадима до 25.05.2026.

### 1.2. Точки синхронизации

| Данные | Направление | Метод | Триггер | Поля |
|--------|-------------|-------|---------|------|
| Расписание врачей | 1С → CRM | Подписка на событие `ПриЗаписиРасписания` | При изменении | Врач, ДатаВремя, Длительность, Статус |
| Номенклатура услуг | 1С → CRM | Запрос при открытии заказа | По требованию | КодУслуги, Наименование, Цена, Категория |
| Факт оказания услуг | 1С → CRM | Подписка на событие `ПриПроведенииУслуги` | При проведении | ПациентID, УслугаID, Дата, Количество |
| Данные оплат | 1С → CRM | Подписка на событие `ПриПроведенииОплаты` | При проведении | ЗаказID, Сумма, ДатаОплаты, МетодОплаты |
| Диагноз (упрощённо) | 1С → CRM | Запрос | По требованию | ТекстРекомендаций (без мед. данных) |
| Сегменты пациентов | CRM → 1С | Реквизиты расширения | При изменении | ТипПациента, ПодтипРеактивации |
| Абонементы | 1С → CRM | Подписка + Регламентное задание | Ежедневно 08:00 | ПациентID, ТипАбонемента, ОстатокЗанятий, ДатаОкончания |

### 1.3. Конфликты данных

| Ситуация | Правило разрешения |
|----------|-------------------|
| Изменение расписания при активной записи | Уведомление администратору + предложение альтернативных слотов |
| Разные суммы в 1С и CRM | Приоритет у 1С; уведомление куратору; OQ-001 |
| Дублирование пациента | Поиск по ФИО + телефон + ДР; предложение объединения |
| Удаление услуги из 1С, на которую ссылается заказ CRM | Логическое помечание устаревшей услуги; уведомление куратору |

## 2. Интеграция с телефонией

### 2.1. Назначение
- Фиксация входящих и исходящих звонков в задачах CRM.
- Определение источника обращения (по номеру телефона).
- Автоматическое создание задачи при входящем звонке от известного пациента.

### 2.2. Требования к интеграции
- API телефонии должен поддерживать вебхуки (событие начала/окончания звонка).
- Передаваемые поля: номер звонящего, номер принимающего, дата/время, длительность, запись разговора (URL или флаг — после юридического согласования, OQ-003).

### 2.3. Неопределённость
> OQ-008: Единая телефония или несколько систем по центрам? Требует ответа Вадима до 30.05.2026. До ответа — интеграция не проектируется.

## 3. Интеграция с мессенджерами (WhatsApp / Telegram)

### 3.1. Назначение
- Отправка авто-сообщений по сценарию дожима (дни 3, 7).
- Напоминания пациентам до визита.
- Получение ответов пациентов (опционально, Фаза 3).

### 3.2. Технические требования

| Параметр | Значение |
|----------|---------|
| WhatsApp | Business API (официальный партнёр) |
| Telegram | Bot API (bot token) |
| Авторизация | API Key / Bearer token |
| Retry | Exponential backoff, max 3 попытки |
| Логирование | Статус отправки (delivered / failed) в CRM_Задача |
| Хранение | Лог отправок 90 дней |

### 3.3. Шаблоны сообщений

Шаблоны хранятся в справочнике `CRM_ШаблоныСообщений`. Поля подстановки: `{Имя}`, `{КомментарийДляПациента}`, `{Телефон}`, `{Куратор}`.

| Шаблон | Использование |
|--------|--------------|
| `DOZIM_DAY3` | Дожим день 3 |
| `DOZIM_DAY7` | Дожим день 7 |
| `REMINDER_1DAY` | Напоминание за 1 день до визита |
| `REMINDER_2DAY` | Напоминание за 2 дня до визита |
| `ABONEMENT_LAST` | Предпоследнее занятие |

## 4. Интеграция с платёжной системой (QR-оплата)

### 4.1. Назначение
Отправка QR-кода для онлайн-оплаты иногородним пациентам и другим случаям.

### 4.2. Требования
- Генерация QR-кода через API платёжной системы (Сбербанк / ЮКасса / Т-Касса — OQ-018).
- Параметры: сумма, комментарий (номер заказа CRM), срок действия.
- Автоматическая фиксация успешного платежа в `CRM_Заказ.СуммаОплачено` через вебхук платёжной системы.
- Синхронизация поступления в 1С:Медицина — через механизм типовых оплат.

### 4.3. Статус
> Фаза 2, задача 2.16. OQ-018 (выбор платёжной системы) — ответ ожидается до 25.06.2026.

## 5. Безопасность и соответствие требованиям

### 5.1. 152-ФЗ (Персональные данные)
- Все персональные данные пациентов хранятся в 1С:Медицина (типовое соответствие ФЗ).
- CRM-расширение не выносит ПДн за пределы информационной базы.
- Логи интеграций: маскирование ФИО, телефонов в записях ошибок.
- Согласие пациента на коммуникации фиксируется в профиле (флаг).

### 5.2. Шифрование
- Транспорт: TLS 1.3 для всех внешних API.
- API-ключи: хранятся в константах 1С (не в коде), доступны только администратору системы.

### 5.3. Аудит
- Все внешние вызовы логируются в регистре `CRM_ЛогИнтеграций`: timestamp, направление, сущность, статус, сообщение об ошибке.
- Retention логов: 90 дней.

## 6. Обработка ошибок

| Ситуация | Действие |
|---------|---------|
| API мессенджера недоступен | Retry 3 раза, затем задача куратору «Связаться вручную» |
| Ошибка отправки QR | Уведомление куратору; запись в лог ошибок |
| 1С:Медицина недоступна (при отдельной базе) | Очередь сообщений; синхронизация при восстановлении |
| Таймаут платёжного API | Повтор через 5 мин; максимум 3 попытки |
