## 1. Что увидено в видеозаписи

### 1.1 Список «Задачи» (форма списка)

Иерархия: **БизнесПроцесс → Задача → Карта маршрута**.

**Командная панель:**
- `К исполнению` — переход в форму выполнения
- `Перенаправить...` — смена исполнителя без потери истории
- `Поручить...` — делегирование
- `Создать на основании ▼` — из задачи можно породить другие документы
- `Ещё ▼` — стандартное меню БСП

**Фильтры верха:**
- **Бизнес-процесс** (Точка маршрута)
- **Пользователь** (Автор / Исполнитель / Все задачи)
- Поле поиска

**Колонки:**
- Задача | Записана (дата/время) | Исполнитель | Срок | Выполнена

В демо видны: «Тестовое задание», «Задание пуганов», «Тесто» — обычные документы типа «Задание».

### 1.2 «Мои задачи» (отфильтрованный список)

Урезанная форма для исполнителя:
- Колонки: Номер | Задача | Срок | Записана
- Чек-бокс «Показывать выполненные»
- Группировка `Сгруппировать по: ▼`

### 1.3 Карточка задачи «Задание (создание)»

**Поля:**
- **Задание** — наименование (строка)
- **Важность** — enum: Обычная / Важная / Очень важная
- **Исполнитель** — ссылка на пользователя (▼ выбор + ↗ открыть)
- **Срок** — дата + время (по умолчанию `23:59:59` текущего дня)
- **Предмет** — связанный документ (пусто или ссылка)
- *Большая текстовая область* — описание/инструкция
- **Проверить выполнение** — чек-бокс (true ⇒ требует подтверждения от автора)
- **Автор** — readonly, текущий пользователь
- **Начало** — readonly, момент создания

**Командная панель:**
- `Стартовать и закрыть` — жёлтая, главная (создаёт задачу и запускает её)
- `Записать` — сохранить без старта
- `Старт` — отдельный запуск
- `▼` (доп. меню) | `Задачи` (родительские) | `Файлы` (вложения)

После сохранения форма меняет заголовок: **«Тесто от 19.05.2026 10:45:32 (Задание)»** — формат `<Имя> от <Дата> <Время> (<ТипОбъекта>)`.

**Изменив форму с очисткой → перенос сроков выполнения и проверки задачи не будут отражены в данной задаче.** — стандартное предупреждение БСП при правке уже стартованной задачи.

### 1.4 «Карта маршрута бизнес-процесса»

**Визуальная схема процесса:**
- Шапка: `Бизнес-процесс` (ссылка) | `Автор` | `Дата` | признак `Стартовая, активная`
- Кнопка `Показать задачи для точки маршрута` — drill-down из узла в список задач
- Графическое поле с точками маршрута (узлы), стрелками переходов, ветвлениями `Да/Нет`

**Пример процесса «Задание (пуганов)»:**
```
постановка задачи
       ↓
[Биленкий Андрей · Выполнить]   ← активная точка (красная рамка)
       ↓
   Нужна проверка?
       ↓ Да         ↓ Нет
   Проверить    Завершение
       ↓
   Провести
       ↓
   Решить
       ↓ Да         ↓ Нет
   Вернуть      Завершение
   исполнителю
```

**Пример процесса «Согласование назначений»:**
```
                     Старт
                       ↓
            Проверить параметры согласования
                       ↓
             ┌─────────┴─────────┐
   Есть назначения              Есть лек.
   услуг?                        назначения?
   ↓ Да      ↓ Нет          ↓ Да       ↓ Нет
   Согласовать              Согласовать
   назначения               лек. назначения
                       ↓
            Обработать результат согласования
                       ↓
                Документ согласован?
                   ↓ Да      ↓ Нет
                            Ознакомиться
                            с результатами
                                 ↓
                         Согласовать повторно?
                            ↓ Да      ↓ Нет
                                      Завершение
```

### 1.5 Связь с конфигурацией

В точках маршрута есть **типизированные действия** (имя метаданных): `БизнесПроцессы.СогласованиеРедактированиеНомерЗаявкиДляПроцесса.БизнесЗадача` — это атрибут БП-точки, ссылающийся на конкретный документ-задачу.

В Δ4 (и далее) под капотом считаем:
- **БизнесПроцесс** = справочник/документ верхнего уровня с маршрутом
- **Задача** (`Задание`) = плоский документ, ссылающийся на родительский БП и на «Точку маршрута» (route point ID)
- **Точка маршрута** = узел на карте, генерирующий задачи при достижении

## 3. Корректировки в FR

### FR-006 (Система задач) — расширить ✏️

Добавить обязательные поля:
- `важность ∈ {Обычная, Важная, Очень важная}` (стало 4-уровневое: enum + `overdue` как состояние)
- `автор` — пользователь, создавший задачу (readonly после сохранения)
- `начало` — datetime создания (readonly)
- `срок` — date + time (по умолчанию `23:59:59`)
- `проверитьВыполнение` (Boolean) — если true, после закрытия задача переходит в состояние «На проверке» к автору
- `файлы[]` — вложения (массив, для прототипа — заглушка)

Изменение терминологии: **«Записать и закрыть» → «Стартовать и закрыть»** для соответствия БСП.

### FR-010 (Стратегия дожима) — формализовать как БП ✏️

Стратегия 0-3-7-14-30 сейчас просто batch-создаёт 5 задач. По паттерну 1С это должен быть **БизнесПроцесс «Дожим пациента»** с картой маршрута:

```
Старт → [День 0 · Звонок куратора] → ответ получен? 
  Да → Завершение
  Нет → [День 3 · WhatsApp] → ответ получен?
    Да → Завершение
    Нет → [День 7 · Эскалация РОП] → решение РОП?
      Продолжить → [День 14 · Повторный WhatsApp] → ...
      Закрыть → Завершение
      В реактивацию → [День 30 · Реактивация] → Завершение
```

Каждая «коробочка» в маршруте — точка маршрута, генерирующая задачу.

### FR-007 (Триггер +3 дня) — формализовать как БП ✏️

Создаётся одиночная задача в рамках БП «Контакт после первичного приёма». В будущем БП может расширяться (доп. этапы).

### FR-008 (Финалка) — формализовать как БП ✏️

«Заполнение блока Ведение» — старт БП «Ведение пациента»:
```
Старт → Сбор данных → 6 обязательных полей валидны?
  Да → [Авто-задача -7 дней до контакта] → [Авто-задача на дату контакта]
       → Контакт состоялся?
         Да → Завершение
         Нет → [Эскалация куратору]
  Нет → Возврат на «Сбор данных» (блок)
```

### FR-014 (Замещение) — учесть БП-маршрут ✏️

При перенаправлении задачи через «Перенаправить...» — в карте маршрута замещаемой задачи фиксируется новая точка «Замещение» с указанием периода.

## 4. Изменения в прототипе (Δ4 scope)

### 4.1 Карточка задачи — переработка

Привести к паттерну 1С:
- **Командная панель:** `Стартовать и закрыть` | `Записать` | `Старт` | `Перенаправить...` | `Поручить...` | `Создать на основании ▼` | `Задачи` | `Файлы`
- **Header:** `<Заголовок> от <Дата> <Время> (Задание)`
- **Поля основной группы:**
  - Задание (textarea, h=22px)
  - Важность (select: Обычная / Важная / Очень важная)
  - Исполнитель (▼ ↗)
  - Срок (date + time, по умолчанию `23:59:59`)
  - Предмет (read-only ссылка на заказ или пациента)
- **Body:** большая textarea с описанием
- **Чекбокс «Проверить выполнение»** + поле «Проверяющий»
- **Footer:** Автор (readonly) + Начало (readonly)

### 4.2 Новый экран «Бизнес-процессы»

Список БП с колонками:
- Номер | Тип процесса | Предмет | Стартован | Активен | Завершён | Автор

Типы процессов:
- «Стратегия дожима» (0-3-7-14-30)
- «Контакт +3 дня после первичного» (FR-007)
- «Ведение пациента» (FR-008)
- «Деактивация куратора» (FR-015)
- «Контроль абонемента» (FR-016)

### 4.3 Новый экран «Карта маршрута БП»

Визуальная схема процесса с:
- Узлами (точки маршрута)
- Стрелками переходов (с условиями Да/Нет)
- Активной точкой (красная рамка)
- Выполненными точками (зелёная заливка)
- Кнопкой «Показать задачи для точки маршрута»

### 4.4 Расширение списка задач

В списке «Задачи кураторов»:
- Колонка «Бизнес-процесс» с ссылкой на родительский БП
- Колонка «Точка маршрута»
- Фильтр «По бизнес-процессу»
